\subsection{Task 2: Coordinate System Transformation}

Building on the code from Lab 1, we added functions for translation and rotation in Lab 2, enabling the aircraft model to change according to the input distance and Euler angles.

To perform \textbf{translation}, we use vector addition to uniformly increase or decrease the position of every vertex on the aircraft. The inputs are $p_n$, $p_e$, and $p_d$, corresponding to position changes in the NED frame.

To perform \textbf{rotation}, we use a NED-to-body-frame transformation with the 3-2-1 sequence rotation matrix. The program first calculates the three individual rotation matrices, then combines them to form the complete rotation matrix $R^b_v$. Each vertex is then multiplied by this matrix. The inputs are $\phi$, $\theta$, and $\psi$, corresponding to roll, pitch, and yaw angles.

\subsection{Task 4: Use Control Surfaces for Aircraft Control}

In the early stages of UAV flight simulation development, it is common to model the aircraft dynamics by directly prescribing the external forces and moments as system inputs. While mathematically convenient, this approach presents a fundamental limitation: a pilot or autopilot does not command forces directly, they deflect physical control surfaces such as the elevator, ailerons, rudder, and throttle.

A more complete simulation framework introduces control surface deflections as the primary input. The aerodynamic forces and moments are computed internally through a \textit{Control Surface} block, which takes the current deflections, aircraft state, and atmospheric conditions, and produces the corresponding forces and moments. This allows the model to capture the full nonlinear coupling between control inputs and vehicle response, including adverse yaw, pitch-roll interaction, and control surface saturation.

\subsection{Task 5: Compute Design Models (Trim)}

The trim model finds the exact combination of aircraft states $\mathbf{x}$ and control inputs $\mathbf{u}$ where the aircraft flies in a steady, non-accelerating equilibrium.

Trim does not calculate deflections with a direct formula. Instead, it solves a constrained nonlinear optimisation problem: find the state $\mathbf{x}$ and controls $\mathbf{u}$ such that, when fed through the full physics model, all accelerations are zero.

\subsection{Task 6: Autopilot Design}

In Lab 6, we design a successive loop closure autopilot based on all previous work. Successive loop closure consists of a series of simple feedback loops that separates time scales: the inner loop handles faster attitude stabilisation, while the outer loop handles navigation.

\subsubsection*{Lateral Autopilot}

The lateral autopilot steers the aircraft left and right through two nested loops:

\begin{itemize}
    \item \textbf{Course control (Controller 2)} --- outer loop PI controller. Computes the error between desired and actual heading and calculates the commanded roll angle $\phi^c$.
    \item \textbf{Roll control (Controller 1)} --- inner loop PD controller. Compares commanded and actual roll angle, then calculates aileron deflection $\delta_a$.
\end{itemize}

\subsection{Model Comparison}

The three models serve different but complementary purposes:

The \textbf{Forces and Moments model} describes how the aircraft's states change over time and is the most intuitive for general audiences, as it directly shows the motion response to applied forces.

The \textbf{Control Surface model} explains the physical causes of motion by computing aerodynamic, gravitational, and propulsive loads, making it most valuable for aviation engineers but harder to understand for general audiences unfamiliar with control surface aerodynamics.

The \textbf{Trim model} determines steady-state equilibrium conditions for a specified airspeed, flight-path angle, and turn radius. It is especially useful for pilots, instructors, and control engineers as it defines the control settings required for balanced flight.

In summary: the forces and moments model is strongest for transient behaviour, the control surface model for physical interpretation, and the trim model for equilibrium analysis. However, trim alone cannot describe manoeuvre response, and none of the models is fully meaningful unless the parameters are accurate.

% ── 4. Concluding Remarks ─────────────────────────────────────────────────────
\section{Concluding Remarks}

\begin{enumerate}

\item \textbf{Key Takeaways}

The project demonstrates a highly effective and adaptable system architecture known as \textit{successive loop closure}. This design manages complex operations by breaking them into simple feedback loops based on a separation of time scales.

\begin{itemize}
    \item \textbf{Inner Loop (Reflexes):} Processes with a faster response time. In flight simulation, the inner loop handles immediate attitude stabilisation by calculating precise control surface deflections.
    \item \textbf{Outer Loop (Strategy):} Responsible for overarching goals that require longer time to execute. In the autopilot, the outer loop governs navigation and course control, issuing high-level commands that the inner loop physically executes.
\end{itemize}

This architecture also maps naturally onto AI agent design. \textit{ReAct} resembles the inner loop --- mixing reasoning and acting in a cycle of thought, action, and observation for fast corrective execution. \textit{OpenClaw's ``Heartbeat''} runs periodic agent turns with full session context, resembling a slower outer monitoring loop. \textit{``Dreaming''} is a background memory consolidation system analogous to long-term adaptation.

\item \textbf{Conclusion}

This project successfully developed a comprehensive MATLAB and Simulink model to simulate and control a fixed-wing aircraft. The implementation and comparative analysis of three distinct flight models --- the Forces and Moments model for transient behaviour, the Control Surface model for physical interactions and nonlinear coupling, and the Trim model for steady-state equilibrium --- provide a complete simulation pipeline from first principles to autopilot flight.

\end{enumerate}
